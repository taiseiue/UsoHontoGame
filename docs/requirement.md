## ウソホントゲーム運営Webアプリケーション 要件定義書

**文書番号:** PM-20251105-001
**作成日:** 2025年11月5日
**作成者:** (あなたの一流のPM/SE)

### 1\. 概要

#### 1.1. プロジェクトの目的

本プロジェクトの目的は、チームビルディングや懇親会等のイベントで利用する「ウソホントゲーム（Two Truths and a Lie）」を、円滑かつインタラクティブに運営するためのWebアプリケーションを開発することである。
手動での集計や進行に伴うオペレーション負荷を軽減し、参加者全員がゲームに集中できる環境を提供すると同時に、リアルタイムのスコア表示によってゲームの盛り上がりを最大化する。

#### 1.2. システムの対象範囲

本システムは、ゲームの事前準備（チーム分け、エピソード登録）から、ゲーム中の進行管理（出題、回答、集計）、最終結果の発表まで、ゲーム運営に関わる一連のプロセスを対象とする。

### 2\. 利用者（ロール）定義

本システムの利用者は、以下の2つのロールに大別される。

| ロール名 | 説明 |
| :--- | :--- |
| **司会者 (Host)** | ゲームイベント全体を管理・進行する責任者。ゲームの作成、チーム設定、ゲーム開始・終了の操作、正解発表など、特権的な操作権限を持つ。 |
| **参加者 (Player)** | ゲームに参加する一般ユーザー。チームに所属し、出題者（Presenter）または回答者（Guesser）としてゲームに参加する。 |

### 3\. 機能要件

ゲームの進行フローに基づき、各ロールに必要な機能を定義する。

#### 3.1. 共通基盤機能

すべての利用者に共通する基本的な機能。

* **認証・参加機能:**
    * ゲームセッション（部屋）への参加機能（例：セッションIDとニックネーム入力による簡易認証）。
    * 司会者専用のログイン機能（管理者権限の付与）。
* **ダッシュボード（メイン画面）:**
    * 現在のゲーム進行状況（「Aチームが出題中」「Bチームが回答中」など）をリアルタイムで表示する。
    * リアルタイム・スコアボード（チーム別得点一覧）を表示する。

#### 3.2. ゲーム準備機能（主に司会者）

ゲーム開始前に必要な設定機能。

* **ゲームセッション管理機能 (司会者):**
    * 新規ゲームセッションを作成する。
    * 参加用URLまたはセッションIDを発行する。
* **チーム管理機能 (司会者):**
    * チーム（例：A〜E）を作成、編集、削除する。
    * 参加者を各チームに割り当てる（ドラッグ＆ドロップなど直感的なUI）。
    * （*オプション:* 参加者が自分でチームを選択できる機能）。
* **ゲーム設定機能 (司会者):**
    * 得点ルール（正解時、騙した時）を設定・確認できる（初期値は10pt / 5pt）。
    * 出題順序を決定する（例：A→B→C...）。

#### 3.3. 出題フェーズ機能

各チームがエピソードを発表する際の機能。

* **エピソード登録機能 (参加者):**
    * 全参加者が、自分が代表者（Presenter）になった場合に備え、3つのエピソード（テキスト）を事前に入力・保存できる。
    * 入力した3つのエピソードのうち、どれが「ウソ(Lie)」であるかをシステムに登録（選択）できる（この正解情報は他の参加者には非公開となる）。
* **出題進行機能 (司会者):**
    * 現在の出題チーム（および代表者）を指定する。
    * 代表者が登録した3つのエピソードを、全参加者の画面に表示させる（例：「1. XXX」「2. YYY」「3. ZZZ」）。

#### 3.4. 回答フェーズ機能

出題チーム以外のチームが「ウソ」を当てる際の機能。

* **回答（相談）時間管理機能 (司会者):**
    * 回答（相談）タイムを開始する。
    * 制限時間のタイマーを設定・起動・表示する。
    * 回答を締め切る。
* **チーム回答投票機能 (参加者):**
    * 各チーム（出題チームを除く）が、チーム内で相談の上、「ウソ」だと思うエピソード番号を1つだけ選択し、投票（送信）できる。
    * （*仕様検討事項:* 投票はチーム代表者1名が行うか、チームメンバーの多数決で自動決定するか。シンプルさを優先し「チーム代表者1名（またはチーム内で最初に入力した人）の回答を採用」を推奨）
* **回答状況表示機能 (司会者):**
    * 各チームが投票済みか否かをリアルタイムで確認できる。

#### 3.5. 結果発表・ポイント集計機能

各ターンの答え合わせとポイント計算機能。

* **正解発表機能 (司会者):**
    * 司会者が「正解発表」ボタンを押す。
* **結果表示機能 (システム/全参加者):**
    * システムが、出題者によって事前に登録された「ウソ」の正解番号を表示する。
    * 同時に、各回答チームがどの番号に投票したかを一覧で表示する。
* **ポイント自動計算機能 (システム):**
    * 提示されたルールに基づき、ポイントを即座に自動計算する。
        * **回答チーム:** 正解した場合、当該チームに +10ポイント。
        * **出題チーム:** 他チームが不正解だった場合、間違えたチーム数 × 5ポイント。
* **スコアボード更新機能 (システム):**
    * 計算結果を即座に全参加者のスコアボードに反映する。

#### 3.6. ゲーム終了機能

全チームの出題が完了した後の機能。

* **ゲーム終了管理機能 (司会者):**
    * 全チームの出題が完了した後、司会者が「ゲーム終了」を宣言する。
* **最終結果表示機能 (システム/全参加者):**
    * 最終的なチームランキング（順位）と合計ポイントを表示する。
    * 優勝チームをハイライトし、祝賀メッセージや簡易的なエフェクト（紙吹雪など）を表示する。

### 4\. 非機能要件

システムの品質に関する要件。

| 項目 | 要件 |
| :--- | :--- |
| **ユーザビリティ (UI/UX)** | \* ゲームのルールを知らない人でも、画面の指示に従うだけで直感的に操作できること。<br>\* 現在の進行状況、自分のチーム、得点状況が常に視覚的に明確であること。<br>\* スマートフォン、タブレット、PCなど、マルチデバイスに対応したレスポンシブデザインであること。 |
| **パフォーマンス (応答性)** | \* 回答の集計、ポイント計算、スコアボードへの反映は、3秒以内に完了すること（リアルタイム性を担保）。<br>\* タイマーの表示に遅延が発生しないこと。 |
| **信頼性・可用性** | \* ゲームイベント中にシステムが停止しないこと（例：想定されるイベント時間中、99.9%の稼働率）。<br>\* 万が一ネットワークが瞬断しても、ゲームの進行状況（スコア、現在のターン）が失われないよう、状態をサーバ側で適切に保持すること。 |
| **拡張性** | \* 将来的にポイント配分（10pt, 5pt）を変更できるように、設定値は定数ではなく管理画面等で変更可能な設計とすること。<br>\* チーム数の増減に柔軟に対応できること。 |
| **セキュリティ** | \* 「ウソ」の正解情報は、正解発表時まで司会者を含む他の全ユーザーから秘匿されること。<br>\* 他チームの回答を（投票完了まで）閲覧できないこと。 |
